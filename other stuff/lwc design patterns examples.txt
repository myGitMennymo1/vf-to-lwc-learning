To keep this practical, I have condensed these into **3 core examples** that cover all the main patterns (Communication, Data, and Composition).

You can copy these into your VS Code, but remember: **each component needs its own folder** in `force-app/main/default/lwc/`.

### 1. The "Service" & "Composition" Pattern

**Goal:** Create a JavaScript-only component (no HTML) that other components can use for logic.

**Folder name:** `utils`
**File:** `utils.js`

```javascript
// No HTML needed for a service component
const formatCurrency = (amount) => {
    return new Intl.NumberFormat('en-US', {
        style: 'currency',
        currency: 'USD',
    }).format(amount);
};

export { formatCurrency };

```

---

### 2. The "Communication" & "One-Way Binding" Pattern

**Goal:** A Parent component passing data down to a Child using `@api`.

**Folder name:** `childComponent`
**File:** `childComponent.js`

```javascript
import { LightningElement, api } from 'lwc';
export default class ChildComponent extends LightningElement {
    @api displayMessage = 'Default Message'; // Parent can overwrite this
}

```

**File:** `childComponent.html`

```html
<template>
    <lightning-card title="Child Component">
        <p class="slds-p-around_medium">{displayMessage}</p>
    </lightning-card>
</template>

```

---

### 3. The "Two-Way Binding" & "Event" Pattern

**Goal:** A component that takes user input and "shouts" it up to a parent.

**Folder name:** `parentComponent`
**File:** `parentComponent.js`

```javascript
import { LightningElement } from 'lwc';
import { formatCurrency } from 'c/utils'; // Importing from our Service Component

export default class ParentComponent extends LightningElement {
    userName = 'Salesforce User';
    formattedPrice = formatCurrency(1000);

    handleUserChange(event) {
        // Two-way binding: capturing the input value
        this.userName = event.target.value;
        
        // Dispatching a Custom Event (Child-to-Parent Pattern)
        const nameChangeEvent = new CustomEvent('namechange', {
            detail: this.userName
        });
        this.dispatchEvent(nameChangeEvent);
    }
}

```

**File:** `parentComponent.html`

```html
<template>
    <lightning-card title="Parent Controller">
        <div class="slds-p-around_medium">
            <lightning-input label="Change Name" value={userName} onchange={handleUserChange}></lightning-input>
            
            <p>Formatted Value from Service: {formattedPrice}</p>

            <c-child-component display-message={userName}></c-child-component>
        </div>
    </lightning-card>
</template>

```

**File:** `parentComponent.js-meta.xml`
*(Set this so you can actually see it on your Home Page)*

```xml
<?xml version="1.0" encoding="UTF-8"?>
<LightningComponentBundle xmlns="http://soap.sforce.com/2006/04/metadata">
    <apiVersion>58.0</apiVersion>
    <isExposed>true</isExposed>
    <targets>
        <target>lightning__HomePage</target>
    </targets>
</LightningComponentBundle>

To bring in actual Salesforce data, we use the **Wire Service** and **Imperative Apex** patterns. These allow your component to communicate with the Salesforce database.

### 4. The "Wire Service" Pattern (Reactive Data)

**Goal:** Automatically fetch record data from Salesforce without writing Apex. If the record updates in the database, the component updates automatically.

**Folder name:** `wireRecordExample`
**File:** `wireRecordExample.js`

```javascript
import { LightningElement, api, wire } from 'lwc';
import { getRecord, getFieldValue } from 'lightning/uiRecordApi';
import NAME_FIELD from '@salesforce/schema/Account.Name';

export default class WireRecordExample extends LightningElement {
    @api recordId; // Automatically populated if on a Record Page

    @wire(getRecord, { recordId: '$recordId', fields: [NAME_FIELD] })
    account;

    get name() {
        return getFieldValue(this.account.data, NAME_FIELD);
    }
}

```

**File:** `wireRecordExample.html`

```html
<template>
    <lightning-card title="Wire Service Pattern">
        <div class="slds-p-around_medium">
            <template lwc:if={account.data}>
                <p>Account Name: {name}</p>
            </template>
        </div>
    </lightning-card>
</template>

```

### 5. The "Imperative Apex" Pattern (On-Demand Data)

**Goal:** Call an Apex method manually (e.g., when a user clicks a button) rather than automatically.

**Folder name:** `imperativeApexExample`
**File:** `imperativeApexExample.js`

```javascript
import { LightningElement } from 'lwc';
import getAccountList from '@salesforce/apex/AccountController.getAccountList';

export default class ImperativeApexExample extends LightningElement {
    accounts;
    error;

    handleLoad() {
        getAccountList()
            .then((result) => {
                this.accounts = result;
                this.error = undefined;
            })
            .catch((error) => {
                this.error = error;
                this.accounts = undefined;
            });
    }
}

```

**File:** `imperativeApexExample.html`

```html
<template>
    <lightning-card title="Imperative Apex Pattern">
        <div class="slds-p-around_medium">
            <lightning-button label="Load Accounts" onclick={handleLoad}></lightning-button>
            <template lwc:if={accounts}>
                <template for:each={accounts} for:item="acc">
                    <p key={acc.Id}>{acc.Name}</p>
                </template>
            </template>
        </div>
    </lightning-card>
</template>

```

### 6. The "Lightning Message Service" (LMS) Pattern

**Goal:** Communication between two components that are **not** related (not parent/child). For this, you first need to create a "Message Channel" in your project.

**File:** `force-app/main/default/messageChannels/SampleMessageChannel.messageChannel-meta.xml`

```xml
<?xml version="1.0" encoding="UTF-8"?>
<LightningMessageChannel xmlns="http://soap.sforce.com/2006/04/metadata">
    <masterLabel>SampleMessageChannel</masterLabel>
    <isExposed>true</isExposed>
    <lightningMessageFields>
        <fieldName>messageToSend</fieldName>
    </lightningMessageFields>
</LightningMessageChannel>

```

**Publisher Component (JS):**

```javascript
import { LightningElement, wire } from 'lwc';
import { publish, MessageContext } from 'lightning/messageService';
import SAMPLE_MC from '@salesforce/messageChannel/SampleMessageChannel__c';

export default class LmsPublisher extends LightningElement {
    @wire(MessageContext) messageContext;

    handleClick() {
        const payload = { messageToSend: 'Hello from LMS!' };
        publish(this.messageContext, SAMPLE_MC, payload);
    }
}

### 1. The Apex Controller (Backend)

In VS Code, go to `force-app/main/default/classes/` and create a file named `AccountController.cls`.

> **Note:** The `@AuraEnabled(cacheable=true)` annotation is vital. It allows the LWC to call the code and tells Salesforce it's safe to cache the results for better performance.

```java
public with sharing class AccountController {
    @AuraEnabled(cacheable=true)
    public static List<Account> getAccountList() {
        return [SELECT Id, Name, Industry FROM Account LIMIT 10];
    }
}

```

### 2. The "Slot" Pattern (Layout Composition)

This is a powerful UI pattern where a component acts as a "shell" or "wrapper," and the parent decides what goes inside.

**Folder:** `customCard`

**File:** `customCard.html`

```html
<template>
    <div class="slds-box slds-theme_default">
        <div class="header">
            <slot name="title"></slot> 
        </div>
        <div class="content">
            <slot></slot>
        </div>
    </div>
</template>

```

---

### 3. The "Error Boundary" Pattern

This lifecycle hook catches errors from any child component so your whole page doesn't turn into a "Technical Error" screen.

**File:** `parentComponent.js` (Add this method)

```javascript
errorCallback(error, stack) {
    this.error = error;
    console.error('Captured error: ', error, stack);
}

```

### Comparison of All Data Patterns

| Pattern | Best Use Case | Benefit |
| --- | --- | --- |
| **Lightning Data Service (LDS)** | Standard CRUD (Create/Read/Update/Delete) | Zero Apex code required. |
| **@wire Service** | Displaying data on load | Automatic refresh when data changes. |
| **Imperative Apex** | Complex logic, Search, or Transactions | Precise control over when the call happens. |
| **Message Service (LMS)** | Cross-app communication | Connects components that aren't related. |
